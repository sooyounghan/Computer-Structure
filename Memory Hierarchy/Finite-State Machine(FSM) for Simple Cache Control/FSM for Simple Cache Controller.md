### 간단한 캐시 제어기를 위한 FSM
-----
1. 간단한 캐시 제어기의 네 가지 상태

<img src="https://github.com/user-attachments/assets/01f2815d-d578-423b-99b9-e3c57ed12475">
</div>

   - 대기 상태(Idle) 상태 : 프로세서에서 유효한 읽기 또는 쓰기 요청이 오기를 기다리는 상태로, 요청이 오면 FSM이 태그 비교 상태로 이동
   - 태그 비교(Compare Tag) 상태 : 이 상태는 읽기나 쓰기 요청이 적중인지 실패인지 검사
     + 주소의 인덱스 부분이 비교할 태그를 선택
     + 인덱스가 가리키는 캐시 블록의 데이터가 유효하고, 주소의 태그 부분이 캐시 블록의 태그와 일치하면 적중
     + 적재 명령어이면 선택된 워드에서 데이터를 읽고, 저장 명령어이면 선택된 워드에 씀 : 그리고 캐시 준비(Cache Ready) 신호가 인가됨
     + 쓰기 요청이면 갱신 비트를 1로 만들며, 쓰기 적중은 유효 비트와 태그 필드 값도 새로 씀 (태그가 단일 메모리이므로 갱신 비트를 바꾸기 위해서 유효 비트와 태그 필드도 새로 써야 함)
     + 만일 적중이고 블록이 유효하다면, FSM은 대기 상태로 돌아감
     + 실패이면 우선 캐시 태그를 갱신하고, 이 블록의 갱신 비트가 1이면 나중 쓰기 상태로, 0이면 할당 상태로 이동
   - 나중 쓰기(Write-Back) 상태 : 이 상태는 태그와 캐시 인덱스로 이루어진 주소를 사용하여 128비트 블록을 메모리에 씀
     + 메모리에서 준비(Ready) 신호가 올 때까지 이 상태에서 기다림
     + 메모리 쓰기가 완료되면 FSM은 할당 상태로 이동
   - 할당(Allocate) 상태 : 메모리에서 새로운 블록을 읽어오며, 메모리에서 준비 신호가 올 때까지 이 상태에서 기다림
     + 메모리 읽기가 완료되면 FSM은 태그 비교 상태로 이동
     + 작업을 완료하기 위해 태그 비교 상태를 재사용하는 대신 새로운 상태로 갈 수 있지만, 쓰기 접근 일 때 블록의 해당 워드를 갱신하는 일을 비롯해 많은 부분이 중복되므로 비교 상태로 가는 것이 좋음

2. 이 간단한 모델의 성능을 향상시키기 위해 더 많은 상태 추가 가능
   - 예를 들어 태그 비교 상태는 태그 비교와 캐시 데이터 읽기 또는 쓰기 두 가지 일을 한 클럭 사이클에 하는데, 클럭 사이클 시간을 단축하기 위해 비교와 캐시 접근을 분리할 수 있음
   - 또 다른 최적화는 쓰기 버퍼를 추가하는 것으로, 쓰기 버퍼에 갱신 블록을 저장하고나서 바로 새로운 블럭을 읽어서, 갱신 블록에 대한 실패가 발생했을 때 두 번의 메모리 접근 동안 프로세서가 기다릴 필요가 없게 함
   - 프로세서가 읽어온 데이터를 이용해 실행하는 동안, 캐시는 쓰기 버퍼의 갱신 블록을 메모리에 씀
