### 명령어를 통한 병렬성
-----
1. 파이프라이닝은 명령어들 사이의 병렬성을 이용
   - 이 같은 병렬성을 명령어 수준 병렬성(Instruction-Level Parallelism, ILP)이라고 함
   - 명령어 수준 병렬성의 양을 증가시키는 두 가지 기본적인 방법
     + 파이프라인의 깊이를 증가시켜 더 많은 명령어들을 중첩시키는 것
       * 세탁소에 비유하자면, 세탁기 사이클이 다른 것들에 비해 길다면, 세탁기를 3개의 기계로 나눌 수 있고, 네 단계 파이프라인에서 여섯 단게의 파이프라인으로 바뀜
       * 최고의 성능을 얻기 위해서는 나머지 단계들을 다시 균형을 잡아서 단계들이 같은 길이를 갖도록 할 필요가 있음
       * 이는 프로세서도 동일하며, 추구되는 병렬성의 양은 늘어남 : 더 많은 연산들이 중첩되기 때문이며, 클럭 사이클이 더 짧아질 수도 있기 때문에, 성능은 더 좋아질 가능성이 있음
     + 컴퓨터 내부의 구성 요소들을 여러 벌 갖도록 하여 매 파이프라인 단게에서 다수의 명령어를 내보낼 수 있도록 하는 것
       * 이 같은 기법의 일반적인 이름을 다중 내보내기(Multiple Issue)라고 함
       * 다중 내보내기 : 한 클럭 사이클에 여러 명령어가 시작되는 기법
       * 말하자면, 다중 내보내기 세탁소에는 세탁기가 3대, 건조기가 3대 존재하며, 같은 시간 안에 세탁물을 접어서 다른 곳에 옮겨 놓는 것도 3배 더 해야 하므로, 보조원도 더 많이 고용해야 할 것
       * 모든 기계를 계속 바쁘게 만들고, 일거리를 다음 파이프라인 단계로 넘기기 위해서 해야 되는 일이 늘어난다는 것이 단점

2. 매 단계마다 다수의 명령을 내보내면 명령어 실행 속도가 클럭 속도보다 빨라질 수 있음
   - 다른 말로 하면, CPI가 1보다 작아질 수 있으며, CPI 값이 1보다 작아지면 척도를 뒤집어서 IPC, 즉, 클럭 사이클 당 명령어 수(Instructions Per Clock Cycle)를 사용하기도 함
   - 따라서, 4 GHz의 4개 명령어 다중 내보내기 마이크로프로세서는 초당 최대 160억개 명령어를 실행할 수 있음
     + 이 경우에는 0.25 CPI 또는 4 IPC를 갖게 되며, 다섯 단계의 파이프라인을 가정하면 그러한 프로세서는 주어진 시간에 20개의 명령어를 동시에 실행하고 있음
     + 오늘날 고성능 프로세서들은 매 클럭 사이클에 3 ~ 6개의 명령어를 내보내려 노력함
   - 그러나 보통은 동시에 실행될 수 있는 명령어 종류에 제약이 있고, 종속성이 있을 때 할 수 있는 일에 대해서도 많은 제약이 있음

3. 다중 내보내기 프로세서를 구현하는 데에는 두 가지 주요 방법이 존재
   - 주된 차이는 컴파일러가 할 일과 하드웨어가 할 일을 나누는 방법
   - 일을 어떻게 나누느냐 것 자체가 여러 가지 결정들이 정적으로 이루어질 것이냐(즉, 컴파일 시) 아니면 동적으로 아루어질 것이냐(즉, 실행 중에)를 결정하기 때문에, 이러한 방법은 때로는 정적 다중 내보내기(Static Multiple Issue)와 동적 다중 내보내기(Dynamic Multiple Issue)라 불림
     + 정적 다중 내보내기 : 많은 결정들이 실행 전에 컴파일러에 의해 이루어지는 다중 내보내기 프로세서 구현 방법
     + 동적 다중 내보내기 : 많은 결정들이 실행 중 프로세서에 의해 이루어지는 다중 내보내기 프로세서 구현 방법
   - 다중 내보내기 파이프라인이 다루어야 할 두 가지 중요한 문제
     + 명령어로 내보내기 슬롯(Issue Slot, 주어진 클럭 사이클에서 명령어들이 내보내질 수 있는 위치, 비유를 들자면 단거리 선수를 위한 출발선 블록의 위치와 동일)을 채우기
       * 대부분 정적 내보내기 프로세서의 경우 적어도 부분적으로 컴파일러가 주어진 클럭 사이클에 얼마나 많은 명령어를 내보내고, 또 어떤 명령어들을 내보낼 것인지 결정
       * 단, 컴파일러가 다중 내보내기에 유리하게 명령어의 순서를 미리 바꾸어서 내보내기 속도를 개선할 수 있게 도와주는 경우가 많음

     + 데이터 해저드와 제어 해저드의 처리
       * 정적 내보내기 프로세서에서는 데이터 해저드와 제어 해저드 전부 또는 일부를 컴파일러가 정적으로 처리
       * 반면에 대부분 동적 내보내기 프로세서는 실행 시간에 작동하는 하드웨어 기술을 이용해 일부의 해저드라도 없애려고 노력

   - 이 두가지가 서로 다른 방법이라고 말하기는 하지만, 실제로는 한 방법이 다른 방법에 차용되는 경우가 많으므로, 어느 방법도 완전히 순수하다고 말할 수 없음
