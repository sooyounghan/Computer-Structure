### 제어 해저드
-----
1. 분기 명령어에 대한 파이프라인 영향
<div align="center">
<img src="https://github.com/user-attachments/assets/0a30c3a5-820e-407d-aa82-78762374c650">
</div>

   - 파이프라인을 유지하기 위해서는 매 클럭마다 명령어를 인출하여야 하는데, 분기를 할 것인지 또는 말 것인지에 대한 결정을 여기서는 MEM 파이프라인 단계에서 이루어짐
   - 이처럼 인출할 명령어가 늦게 결정되는 문제를 제어 해저드 또는 분기 해저드라고 부름
   - 제어 해저드는 데이터 해저드만큼 자주 일어나지 않고, 또 데이터 해저드의 전방전달처럼 효과적인 해결책이 없음

2. 분기가 일어나지 않는다고 가정
   - 분기가 끝날 때까지 파이프라인을 지연시키는 것은 너무 느림
   - 분기 지연(Branch Stalling)보다 더 좋은 방법 하나는 분기가 일어나지 않는다고 예측하고, 명령어들을 순서대로 실행하는 것
   - 만약, 분기가 일어난다면 인출되고 해독되었던 명령어들을 버리고 분기 목적지에서 실행을 계속함
     + 조건부 분기의 반 정도는 분기하지 않고, 명령어를 버리는데 드는 비용이 거의 없다면, 이 최적화 방법으로 제어 해저드 대가를 반으로 줄일 수 있음
     + 명령어를 버리기 위해서는 적재-사용 데이터 해저드의 경우 지연시키기 위해 했던 것과 같이 원래의 제어값을 0으로 바꾸면 됨
     + 분기 명령어가 MEM 단계에 도달했을 때, IF / ID / EX 단계에 있는 명령어 3개를 바꿔야 한다는 것
     + 적재-사용 지연의 경우, ID 단계의 제어값만을 0으로 바꾸어 파이프라인을 통해 천천히 지나가도록 함
     + 명령어를 버린다는 것은 파이프라인의 IF / ID / EX 단계에 있는 명령어를 쓸어내야(Flush) 한다는 것을 의미 (쓸어내기(Flush) : 파이프라인에 있는 명령어들을 버리는 것으로, 보통 예상하지 않았던 사건때문에 시행)
