### 창고 규모의 컴퓨터
-----
1. WSC에서 일괄 처리(Btach Processing)를 위해 널리 알려진 프레임워크로 MapReduce와 공개 소프트웨어인 Hadoop이 존재
   - 같은 이름을 가진 Lisp 함수에 착안해서 만든 Map은 프로그래머가 작성한 함수를 각각의 논리적 입력 레코드에 적용
   - Map은 수천 개의 서버를 사용하여 키-값(Key-Value) 쌍 형태의 중간 결과를 생성
   - Reduce는 이렇게 분산 수행된 태스크 결과값을 모아 프로그래머가 정의한 또 다른 함수를 사용해 통합된 결과 산출
   - 적절한 소프트웨어 지원을 받으면 Map과 Reduce는 이해하기 쉬우며, 사용하기도 쉽고 매우 병렬적임

2. 예) 문서의 모음에서 각 영어 단어가 등장하는 빈도수를 계산하는 Map Reduce 프로그램
   - 어떤 한 문서에서 모든 영어 단어가 딱 한 번씩 등장한다는 것을 가정하고 작성한 프로그램의 내부 수ㅜㄴ환문

   - Map 함수에 사용하는 EmitIntermediate 함수는 문서에 등장하는 각 단어와 1을 출력
   - 그리고 Reduce 함수는 ParseInt 함수를 이용해 문서 내 각 단어의 값을 모두 더해 모든 단어의 출현 빈도수를 구함
   - MapReduce 런타임 환경이 map 태스크와 reduce 태스크를 WSC 서버에 스케줄

3. WSC는 1970년대 슈퍼컴퓨터를 극단적으로 크게 만들어서 전력 공급과 냉각, 모니터링, 운영 면에서 혁신을 불러온 것
4. WSC는 서버와 공통적 목표를 몇 개 가지고 있지만, 다음 세 가지 면에서 서버와 다름
   - 대규모이면서 쉬운 병렬성
     + Map, Reduce와 같은 일괄처리 응용은 웹 크롤(Web Crawl) 응용에서 수십억개의 페이지를 탐색하는 것처럼, 독립적인 처리가 필요한 대량의 독립적인 데이터가 있을 때 유리
     + 서비스로의 소프트웨어, 즉 SaaS(Software as a Service, 고객의 컴퓨터에 장착되어 수행되는 소프트웨어를 판매하는 것이 아닌, 소프트웨어는 원격지에서 실행되고 고객은 웹 인터페이스를 통해 인터넷으로 사용, SaaS 고객은 소유가 아닌 사용에 근거해 비용을 지불)로 알려진 쌍방시 인터넷 서비스 응용은 수백만 명의 독립적 사용자가 쌍방식 인터넷 서비스를 사용하는 특징이 있음
       * SaaS에서는 읽기와 쓰기가 의존적인 경우가 거의 없으므로 동기화가 거의 필요 없음
       * 예를 들어, 검색은 읽기 전용 인덱스를 사용하며, 이메일은 서로 독립적인 정보를 읽고 쓰는 것이 보통
       * 이렇게 쉬운 병렬성의 형태를 요구 수준 병렬성(Request-Level parallelism)이라고 칭하는데, 통신과 동기화의 필요성이 거의 없이 자연스럽게 병렬적으로 처리할 수 있는 다수의 독립적 작업들의 행동을 의미

   - 가동 비용의 중요성 : WSC는 수명이 길어서 건물과 전력 및 냉각 설비가 보통 10년 혹은 20년에 걸쳐 감가상각
   - 확장성과 확장에 따른 기회와 문제들 : 하나의 WSC를 구축하기 위해 50,000대 이상 서버와 아울러 이를 지원하는 기반 설비를 구매해야 하므로, 대량 구매에 따른 할인을 받을 수 있음
     + WSC 자체의 규모가 상당하므로 규모의 경제를 누릴 수 있으므로, 클라우드 컴퓨팅이 탄생
     + WSC의 유닛당 단가가 낮으므로, 클라우드 업체들이 적절한 이윤이 날 만큼 서버 임대료를 책정해도 사용자들이 직접 서버를 구매하는 것보다 저렴하게 할 수 있음
     + 규모의 경제 이면에는 규모에 따라 고장 확률이 높아지는 문제가 존재하므로, 고장 감내는 중요한 이슈

5. MapReduce 프레임워크는 Map 단계의 끝 부분에서 키-값 쌍을 뒤섞은 후 정렬에서 같은 키를 갖는 것들을 모아 그룹을 모아 만들고, 이 그룹들을 Reduce 단계로 보냄
6. 대규모 컴퓨팅의 또 다른 형태는 그리드 컴퓨팅(Grid Computing) : 그리드 컴퓨팅에서는 지리적으로 멀리 떨어진 컴퓨터들이 장거리 통신망을 이용해 서로 통신하며 프로그램을 수행
