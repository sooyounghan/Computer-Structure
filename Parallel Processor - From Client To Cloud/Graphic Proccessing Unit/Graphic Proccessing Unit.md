-----
### 그래픽 처리 유닛
-----
1. 기존 구조에 SIMD 명령어를 추가하는 것이 타당하다고 생각하게 된 초창기 이유는 PC와 워크스테이션에 있는 많은 마이크로프로세서들이 그래픽 디스플레이와 연결되어 있어서 그래픽을 처리하는 시간의 비중이 점점 늘어난다는 것
   - 마이크로프로세서에 사용 가능한 트랜지스터 수가 늘어나면서, 그래픽 처리 성능을 개선하는 것이 타당한 일이 됨

2. 그래픽 프로세서는 CPU와 구별하기 위해서 그래픽 처리 유닛(GPU)이라는 이름을 사용하게 됨
3. CPU와 GPU의 주요 차이점
   - GPU와 CPU를 돕는 가속기로, 따라서 CPU가 수행하는 모든 작업을 다 수행할 필요가 없음
     + 그렇디 때문에, 모든 자원을 그래픽만을 위해 사용할 수 있음
     + GPU가 어떤 작업을 전혀 수행할 수 없거나 느리게 수행한다고 해도 그 시스템이 CPU와 GPU를 모두 가지고 있다면 문제되지 않음 : CPU가 그 일을 담당하면 됨
   - GPU 문제의 크기는 보통 수백 MB에서 GB 수준에 이름 : 수백 GB이나 TB까지 가지 않음

4. CPU나 GPU의 차이점으로 인한 다른 스타일 구조
   - 아마 가장 큰 차이점은 GPU의 경우 메모리의 큰 지연시간을 극복하기 위해 CPU처럼 다단계 캐시를 사용하지 않음
     + 그 대신 GPU는 메모리 지연을 감추기 위해 하드웨어 멀티스레딩을 사용 : 즉, 메모리 접근을 요청한 후 그 데이터가 도착할 때까지 GPU는 그 요청과 무관한 스레드를 수백, 수천 개 수행

   - GPU의 메인 메모리는 지연시간보다는 처리율을 높이는 데 초점을 맞춤
     + GPU를 위한 DRAM 칩보다 폭이 더 넓고 처리량이 큰 DRAM 칩을 GPU 전용으로 두기도 함
     + GPU 메모리는 통상 마이크로프로세서의 메인 메모리보다 더 작음
     + GPU는 코프로세서이므로, 범용 연산을 할 때는 CPU 메모리와 GPU 메모리 간의 전송 시간이 추가됨

   - 큰 메모리 대역폭을 얻기 위해서 많은 수의 스레드를 활용하기 때문에 GPU는 많은 스레드 뿐 아니라 많은 병렬 프로세서(MIMD)도 수용할 수 있음 : 따라서, 전형적인 CPU와 비교할 때 GPU는 훨씬 고도로 멀티스레딩되어 있고 프로세서도 더 많음

5. GPU는 좁은 응용 범위를 대상으로 설계되었지만, 일부 프로그래머들은 GPU의 높은 잠재 성능을 이용할 수 있도록 응용 프로그램을 작성할 수 있는지 궁금
   - 처음에는 그래픽 API와 언어를 사용하여 문제를 서술하려 하다가, 직접 GPU 프로그램을 작성할 수 있도록 C 언어에 근간을 둔 프로그램 언어 개발 : NVIDIA사의 CUDA(Compute Unified Deviced Architecture)
   - 일부 제약이 있기는 하지만 프로그래머가 GPU에서 수행될 C 프로그램을 직접 작성할 수 있게 해줌
   - NVIDIA는 여러 형태의 병렬성을 통합하는 주제를 CUDA 스레드로 정의
     + 가장 낮은 수준의 이 병렬성을 프로그래밍 프리미티브로 사용하여, 컴파일러와 하드웨어가 수천 개의 CUDA 스레드를 묶어 멀티스레딩, MIMD, SIMD, 명령어 수준의 병렬성 등 GPU 내 다양한 병렬성을 활용할 수 있음
     + 이 스레드들은 32개씩 묶여져서 블록을 이루고 한꺼번에 수행
     + CPU 안에 있는 멀티스레드 프로세서는 이 스레드 블록들을 실행하는데, GPU는 8 ~ 128개 멀티스레드 프로세서로 구성
